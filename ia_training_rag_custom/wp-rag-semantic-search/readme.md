# RAG Semantic Search — WordPress Plugin

**Version:** 1.0.11
**Author:** Bruno Flaven
**Requires:** WordPress 5.6+, PHP 7.4+, MySQL 8.0+ / MariaDB 10.3+

A WordPress plugin that imports a RAG-processed article corpus and exposes semantic search via a shortcode and a dedicated sidebar widget. Works offline — no external API calls required during development.

---

## What it does

1. **Imports** a JSON exchange file (`rag_bridge.json`) generated by the RAG system into two custom MySQL tables.
2. **Searches** those tables using MySQL FULLTEXT ranking and renders scored result cards.
3. **Caches** every result set in `wp_rag_results` (7-day TTL, keyed by SHA-256 of the query). Repeated searches are served from the cache — no second FULLTEXT query needed.
4. **Offers a mode toggle** on the search page — Standard search (native WP) or Semantic search (this plugin).
5. **Provides a sidebar widget** — drag it into any widget area to put a semantic search box in the sidebar, footer, or anywhere else.
6. **Admin toggle** — enable or disable semantic search from the plugin admin page without touching any code.

## Installation

### Option A — Copy folder (local / Docker)

```bash
cp -r rag-semantic-search/  /path/to/wordpress/wp-content/plugins/
```

> When copying files directly (Docker, SFTP), the WordPress activation hook does not fire.
> The plugin detects this automatically: on the first page load after the copy, it checks
> whether the tables exist and creates them if they are missing. No manual step required.

### Option B — Upload via WP Admin

1. Zip the `rag-semantic-search/` folder.
2. WP Admin → Plugins → Add New → Upload Plugin → choose the zip → Install Now.

### Activate

WP Admin → Plugins → find **RAG Semantic Search** → Activate.

On activation the plugin creates two tables in your WordPress database:

| Table | Purpose |
|---|---|
| `wp_rag_posts` | One row per imported article (id, title, url, date, slug, excerpt, text) |
| `wp_rag_results` | One row per (query, ranked result) pair — TTL-managed cache |

> The `wp_` prefix matches your WordPress installation's `$table_prefix`. If your prefix
> is different (e.g. `blog_`), the tables will be `blog_rag_posts` and `blog_rag_results`.

## Quick start

### Step 1 — Generate the exchange file (RAG side)

From the RAG project root:

```bash
conda activate ia_achats
python bridge_export.py
# Writes: data/bridge/rag_bridge.json
```

### Step 2 — Import posts

WP Admin → **RAG Search** → upload `rag_bridge.json` → **Import posts**.

Expected output: *Import complete — 74 inserted, 0 updated, 0 skipped.*

Re-importing the same file is safe. Existing rows are updated, not duplicated.

### Step 3 — Create the search page

1. WP Admin → Pages → Add New.
2. Title: `Semantic Search` (slug: `semantic-search`).
3. Body: add the shortcode `[rag_search]`.
4. Publish.

### Step 4 — Add the sidebar widget

1. WP Admin → Appearance → Widgets.
2. Find **RAG Semantic Search** in the available widgets list.
3. Drag it into the sidebar (or any widget area).
4. Set the **Title** (e.g. *Semantic Search*).
5. Set the **Results page URL** to the page created in Step 3
   (e.g. `http://localhost:8080/semantic-search/`).
6. Save.

Visitors can now type a query in the sidebar and land directly on the results page.

### Step 5 — Test

- Sidebar widget → type a query → lands on `/semantic-search/?rss_q=your+query`
- On the results page, use the mode toggle to switch between Semantic and Standard search.

## Search modes

| Mode | Behaviour |
|---|---|
| **Semantic search** | Stays on the results page, queries `wp_rag_posts` via MySQL FULLTEXT, renders scored cards. Default mode. |
| **Standard search** | Redirects to `/?s=query` — native WordPress search |

**URL structure:**
- Semantic: `/semantic-search/?rss_q=Thiel`
- Standard: `/?s=Thiel` (redirect handled by the plugin)

**How scores work:** the top result always scores `1.00`. All other results are normalised relative to it. The score bar gives an instant visual sense of how much each result trails the top hit.

**Fallback:** if FULLTEXT returns no results (e.g. query words are under 4 characters), the plugin falls back to a `LIKE` search on title, excerpt, and text.

**Result cache:** every search result set is written to `wp_rag_results` with a 7-day TTL. The next identical query is served directly from the cache — no FULLTEXT query runs. The cache key is the SHA-256 hash of the lowercased, trimmed query string. Expired rows are purged automatically before each search. The admin Status section shows the current cache row count.

## Admin page

WP Admin → **RAG Search** (menu icon: search).

### Status section

Live row counts for both tables — refreshes on every page load.

### Settings section

**Enable semantic search** checkbox — checked by default.

| State | Effect |
|---|---|
| Checked | Full semantic search: mode toggle visible, FULLTEXT ranking active |
| Unchecked | Mode toggle hidden, submit always redirects to native WP search (`/?s=query`) |

Disabling does not delete any data. Re-enabling restores everything instantly.

### Import section

Upload `rag_bridge.json`. The importer:
- Validates file extension (`.json` only) and size (50 MB cap).
- Validates required fields per record: `id`, `title`, `url`, `date`, `slug`, `text`.
- Inserts new records, updates existing ones (`ON DUPLICATE KEY UPDATE`).
- Reports: inserted / updated / skipped / errors.

**Clear before import checkbox** — when checked, all rows in `wp_rag_posts` are deleted before the new file is processed. Use this for a full replacement instead of an upsert. Unchecked by default.

### Export section

**Export posts** — downloads the full contents of `wp_rag_posts` as a `rag_bridge_export_YYYY-MM-DD.json` file. The exported schema is identical to the import format (`id`, `title`, `url`, `date`, `slug`, `excerpt`, `text`), so the file can be edited and re-imported without any transformation. The current post count is shown so you can confirm data is present before exporting.

### Search cache section

**Clear search cache** — deletes all rows from `wp_rag_results` only. Posts in `wp_rag_posts` are untouched. The current cache row count is displayed so you can judge whether a flush is warranted. Use this if the cache has grown unexpectedly large or you want to force fresh FULLTEXT results for all queries.

The cache is also self-managing: expired rows (older than 7 days) are purged automatically before every search, so manual clearing is rarely needed.

### Danger zone

**Empty tables** — deletes all rows from both tables without dropping them. A JS confirmation dialog prevents accidental clicks. Use this to wipe and re-import from scratch.

## Uninstall

WP Admin → Plugins → RAG Semantic Search → **Delete**.

On deletion, `uninstall.php` runs automatically and:
- Drops `wp_rag_results` (results first, due to FK dependency).
- Drops `wp_rag_posts`.
- Removes all plugin options from `wp_options`: `rss_db_version`, `rss_semantic_enabled`, `rss_promo_text`, `rss_lang_notice`, `rss_cache_ttl_days`.

> **Deactivating** the plugin does NOT delete any data. Tables and records are preserved until the plugin is fully deleted.

## Database schema

```sql
-- Posts imported from rag_bridge.json
CREATE TABLE wp_rag_posts (
  id          INT            NOT NULL,        -- WP post ID (primary key)
  title       VARCHAR(500)   NOT NULL,
  url         VARCHAR(1000)  NOT NULL,
  `date`      DATE           NOT NULL,
  slug        VARCHAR(500)   NOT NULL,
  excerpt     TEXT,
  `text`      LONGTEXT       NOT NULL,
  imported_at DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id)
);

-- Search result cache (TTL-managed)
CREATE TABLE wp_rag_results (
  id          INT            NOT NULL AUTO_INCREMENT,
  query_hash  CHAR(64)       NOT NULL,        -- SHA-256 of query_text
  query_text  TEXT           NOT NULL,
  post_id     INT            NOT NULL,        -- FK → wp_rag_posts.id
  score       FLOAT          NOT NULL,
  `rank`      TINYINT        NOT NULL,        -- 1-based position (backtick-escaped: MySQL 8.0 reserved word)
  created_at  DATETIME       NOT NULL DEFAULT CURRENT_TIMESTAMP,
  expires_at  DATETIME       NOT NULL,
  PRIMARY KEY (id),
  KEY idx_query_hash (query_hash),
  KEY idx_expires_at (expires_at)
);
```

> `rank` is a reserved keyword in MySQL 8.0 (window functions). It is backtick-escaped
> in `class-db.php` so `dbDelta()` creates the table correctly.

Full reference (with utility queries): `../data/bridge/rag_schema.sql`

## Exchange format

The plugin imports `rag_bridge.json` — a JSON array generated by `bridge_export.py`.

| Field | Type | Required | Description |
|---|---|---|---|
| `id` | integer | yes | WordPress post ID |
| `title` | string | yes | Plain text title |
| `url` | string | yes | Canonical post URL |
| `date` | string | yes | `YYYY-MM-DD` |
| `slug` | string | yes | URL slug |
| `excerpt` | string | no | Short summary, may be empty |
| `text` | string | yes | Full article body, plain text |

Tags and categories are intentionally excluded (IDs only in source, no label resolution).
Full spec: `../data/bridge/rag_bridge_schema.json`

## Troubleshooting

**Tables not created after copying files (Docker / SFTP)**
- No action needed — the plugin creates tables automatically on the first page load after
  the copy (`plugins_loaded` hook checks `SHOW TABLES` and runs `dbDelta` if missing).
- If tables still do not appear, verify the DB user has `CREATE TABLE` privileges.

**"Unexpected output during activation" warning**
- Fixed in 1.0.2. The `rss_activate()` hook now wraps `dbDelta()` in an output buffer.
  Update to 1.0.2 or later.

**Import shows 0 inserted, 0 updated**
- Verify the file is valid JSON (array at top level, not an object).
- Open browser devtools → Network tab → check the POST response for a PHP error.

**"Search results cached" always shows 0**
- Fixed in 1.0.9. In prior versions `wp_rag_results` was never written to — results were
  discarded after each request. Upgrade to 1.0.9 and run any search once to populate the cache.

**Search returns no results**
- MySQL FULLTEXT minimum word length is 4 by default (`ft_min_word_len`). Queries shorter
  than 4 characters fall back to LIKE search automatically.
- Verify `wp_rag_posts` has rows: WP Admin → RAG Search → Status section.

**Score bar is all the same width**
- This happens when FULLTEXT falls back to LIKE (all scores equal `1.0`).
  Try a longer, more specific query.

**Sidebar widget does not redirect to the right page**
- Check the **Results page URL** in the widget settings (Appearance → Widgets).
  It must match the exact URL of the page containing `[rag_search]`.
